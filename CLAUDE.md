# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

rlottie is a platform-independent, standalone C++ library that renders Adobe After Effects
animations exported as bodymovin JSON (Lottie) to a raw ARGB32-premultiplied pixel buffer. It
ships its own software vector-graphics engine and has **no external runtime dependencies** for
the core library (gtest is only needed for the test suite).

The render pipeline is: **bodymovin JSON → parser → immutable model tree → per-frame renderer
item tree → software rasterizer → `Surface` pixel buffer.**

## Build

Two build systems are kept in sync; **meson** is the primary one.

```sh
# Meson (primary)
meson build
ninja -C build

# CMake
mkdir build && cd build && cmake .. && make -j2
cmake -DBUILD_SHARED_LIBS=OFF ..        # static
cmake -DLOTTIE_ASAN=ON ..               # AddressSanitizer build
```

`config.h` is generated by the build from `cmake/config.h.in` (CMake) or inline (meson). It
controls feature macros: `LOTTIE_THREAD_SUPPORT`, `LOTTIE_CACHE_SUPPORT`,
`LOTTIE_IMAGE_MODULE_SUPPORT` (dlopen'd image-loader plugin for embedded PNG/JPEG),
`LOTTIE_LOGGING_SUPPORT`, `LOTTIE_DUMP_TREE_SUPPORT`. Toggle via meson options
(`meson_options.txt`) or CMake `option()`s (`-Dlog=true`, `-DLOTTIE_THREAD=OFF`, etc.).

Cross builds: `arm_build.sh` (uses `arm_cross.txt`), `wasm_build.sh` (uses `wasm_cross.txt`,
emits the wasm module from `src/wasm/`). Windows MSVC project lives in `vs2019/` and
`example/win32Player/`.

## Test

Tests use **GoogleTest** and are off by default.

```sh
# Meson
meson configure -Dtest=true build
ninja -C build
ninja -C build test

# CMake
cmake -DLOTTIE_TEST=ON ..
make && ctest

# Run a single test (binaries are plain gtest executables)
./build/test/vectorTestSuite --gtest_filter='VPath.*'
./build/test/animationTestSuite --gtest_filter='*loadFromData*'
```

Two suites: `vectorTestSuite` (the `src/vector` engine — `test_vrect.cpp`, `test_vpath.cpp`)
and `animationTestSuite` (the public C++ and C APIs — `test_lottieanimation*.cpp`). Tests load
fixtures from `example/resource/` via the `DEMO_DIR` compile define.
`example/resource/test/repro_*.json` are crash/overflow regression fixtures for the security
hardening that is the current focus of this fork (see recent commits); add a fixture there when
fixing a parser/renderer crash.

## Demo / running

The build produces `lottie2gif`, a JSON→GIF converter usable to eyeball a render:
`lottie2gif path/to/file.json`. Other example binaries (`demo`, `lottieviewer`, etc.) require
EFL/elementary and are skipped when it is absent.

## Architecture

### API surface (`inc/`)
- `rlottie.h` — public **C++** API. `rlottie::Animation` (load + query + render),
  `rlottie::Surface` (caller-owned buffer + draw region), and the templated dynamic-property
  system (`Property` enum + `setValue<Property::X>(keypath, value)`).
- `rlottie_capi.h` — public **C** API (mirrors the C++ one).
- `rlottiecommon.h` — POD structs describing the C render tree (`LOTNode`, `LOTLayerNode`,
  `LOTMask`), returned by `renderTree()` and used by the C binding.

### Source layout (`src/`)
Three independent layers:

- **`src/vector/`** — a self-contained 2D software vector engine (think mini-Skia). Everything
  is `V`-prefixed: `VPath`, `VMatrix`, `VBezier`, `VRle` (run-length encoded coverage), `VRaster`
  (scanline rasterizer), `VBrush`/`VPainter`/`VDrawable`, `VBitmap`, `VDasher`. SIMD blend paths
  in `vdrawhelper_sse2.cpp` / `vdrawhelper_neon.cpp`. Vendored: `freetype/` (the `SW_FT_*`
  raster + stroker), `pixman/` (ARM NEON asm), `stb/` (image decode). Has **no dependency** on
  the lottie layer and is tested standalone. Uses the `V_USE_NAMESPACE` macro.
- **`src/lottie/`** — the Lottie-specific layer. Vendored `rapidjson/` for parsing and `zip/`
  (miniz) for `.lottie`/dotLottie archives.
- **`src/binding/c/`** — `lottieanimation_capi.cpp`, the C API implementation over the C++ one.

### The two model trees (the key mental model)
Code lives under two sibling namespaces inside `rlottie::internal`:

- **`model::`** (`lottiemodel.h/.cpp`, `lottieparser.cpp`) — the **immutable parsed data**. The
  parser turns JSON into a `model::Composition` (layers, shapes, keyframed properties). This is
  what gets **cached and shared** across `Animation` instances.
- **`renderer::`** (`lottieitem.h/.cpp`) — the **mutable per-frame render tree**. A
  `renderer::Composition` wraps a `model::Composition` and, for each frame, evaluates animated
  properties, computes matrices/alpha (dirty-flagged), builds masks/mattes/clips, and rasterizes
  into the surface. Allocations come from a bump `VArenaAlloc`.

`AnimationImpl` (`lottieanimation.cpp`) owns one `renderer::Composition` and drives
`update(frame) → render(surface)`.

### Loading & caching
`lottieloader.cpp` reads files/strings and parses them. A process-global `ModelCache`
(keyed by file path or caller-supplied key) memoizes `model::Composition`s; size is configurable
via `rlottie::configureModelCacheSize()` and disabled with size 0. Per-call caching can be
opted out with the `cachePolicy` argument.

### Threading
`lottieanimation.cpp` contains a work-stealing `RenderTaskScheduler` (one queue per hardware
thread, round-robin dispatch). `Animation::render()` is the async path — it returns a
`std::future<Surface>`; `renderSync()` is the blocking path. A separate `RasterTaskScheduler`
(in `src/vector/vraster.cpp`) parallelizes rasterization. Each `Animation` renders into its own
buffer, so renders of different animations are independent. When `LOTTIE_THREAD_SUPPORT` is off,
both schedulers degrade to synchronous execution.

### Dynamic properties (runtime overrides)
`setValue<Property::X>("layer.group.fill", value)` overrides animation properties at runtime.
A `LOTKeyPath` (`lottiekeypath.cpp`) — supporting `*` (one level) and `**` (globstar) wildcards
— is resolved against the render tree; matches receive a `LOTVariant` filter
(`lottiefiltermodel.h`, `lottieproxymodel.cpp`) that is applied every frame. The value may be a
constant or a `std::function<T(FrameInfo)>` callback.

## Conventions that constrain how you write code here

- **The library is built with exceptions and RTTI disabled** (`-fno-exceptions -fno-rtti`,
  MSVC `/EHs-c- /GR-`). Do not add `throw`, `try/catch`, `dynamic_cast`, or `typeid` to library
  code. Handle errors by return value / early-out, as the existing code does.
- **C++14** for the library; the public headers must stay **C++11**-compatible.
- Symbol visibility is hidden by default; the exported ABI is pinned by `rlottie.expmap`
  (version script) on Linux. New public symbols must be added there.
- Formatting is clang-format (`.clang-format`). The `format` script formats only the last
  commit's diff: `git diff -U0 --no-color HEAD^ | clang-format-diff -i -p1`.
- The vector engine never includes lottie headers — keep that one-way dependency intact.
